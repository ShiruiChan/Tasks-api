## 2. Описание разработанного сервиса

Разработан сервис **Tasks API** - REST API для управления списком задач
(to-do list). Сервис реализует полный набор CRUD-операций.

**Технологический стек:**
- Python 3.11
- FastAPI - веб-фреймворк для построения API
- Pydantic - валидация данных
- Uvicorn / Gunicorn - ASGI-сервер для запуска
- pytest - фреймворк тестирования
- Docker - контейнеризация

**Эндпоинты API:**

| Метод    | Путь          | Описание                   |
|----------|---------------|----------------------------|
| `GET`    | `/health`     | Проверка работоспособности |
| `GET`    | `/tasks`      | Список задач               |
| `POST`   | `/tasks`      | Создать задачу             |
| `GET`    | `/tasks/{id}` | Получить задачу            |
| `PATCH`  | `/tasks/{id}` | Обновить задачу            |
| `DELETE` | `/tasks/{id}` | Удалить задачу             |

## 3. Тестирование (pytest)

Написано **13 тестов**, покрывающих все эндпоинты и сценарии:
- успешное создание, чтение, обновление и удаление задач;
- обработка ошибок валидации (HTTP 422 при пустом названии);
- обработка несуществующих задач (HTTP 404);
- сквозной сценарий работы со списком задач.

Тесты используют `TestClient` из FastAPI и фикстуру с очисткой хранилища
перед каждым тестом (изоляция тестов).

**Результат запуска `pytest`:**

```

tests/test_api.py .............                                          [100%]

```

## 4. Контейнеризация (Docker)

Написан `Dockerfile` на базе образа `python:3.11-slim`. Применены практики:
- кэширование слоёв (зависимости устанавливаются до копирования кода);
- запуск через gunicorn с воркерами uvicorn;
- проброс переменной окружения `$PORT` для совместимости с хостингами.

**Сборка и запуск:**

```bash
docker build -t tasks-api .
docker run -p 8000:8000 tasks-api
```

## 5. Деплой

Сервис развёрнут на хостинге **Render** (бесплатный план).
Деплой настроен через файл `render.yaml` (Blueprint):

- **Build:** `pip install -r requirements.txt`
- **Start:** `gunicorn app.main:app -k uvicorn.workers.UvicornWorker -b 0.0.0.0:$PORT`
- **Health check:** `/health`

Также подготовлены конфигурации для альтернативных способов деплоя:
- `Procfile` - для Railway;
- инструкция для **VPS** (screen + gunicorn + nginx как реверс-прокси).

**Ссылка на демо:** `https://<имя-сервиса>.onrender.com`
*(подставить актуальную ссылку после деплоя)*
